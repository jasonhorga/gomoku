### 胜负判定

每次落子后，从该位置出发检查 4 个方向：

```
方向1: 水平   → (0, +1) 和 (0, -1)
方向2: 垂直   → (+1, 0) 和 (-1, 0)
方向3: 对角↘  → (+1, +1) 和 (-1, -1)
方向4: 对角↙  → (+1, -1) 和 (-1, +1)

示例: 黑子在 (7,7) 落子后检查水平方向
  ← 向左数连续黑子 + 1(自身) + 向右数连续黑子 ≥ 5 → 赢

  . . ● ● ● [●] ● . .     ← 5连，黑赢!
              ↑ 刚落的子
```

时间复杂度：O(1) — 只检查落子位置周围，不扫描整个棋盘。

## 3. Player 抽象层

### 设计模式

```
             GameManager
                 │
        _on_move_decided(row, col)
                 │
     ┌───────────┼───────────┐
     │           │           │
 HumanPlayer  AIPlayer   NetworkPlayer
     │           │           │
  鼠标点击    AI计算(Thread)  RPC网络
```

核心思想：**GameManager 不关心落子从哪来**。它只管理两个 PlayerController 实例，
调用 `request_move()` 然后等 `move_decided` 信号。这样添加新的落子来源（AI、网络、
甚至文件回放）都不需要改 GameManager 的代码。

### 游戏循环

```
start_game()
    │
    ▼
_request_current_move()  ←──────────┐
    │                               │
    ▼                               │
 [等待 move_decided 信号]            │
    │                               │
    ▼                               │
_on_move_decided(row, col)          │
    │                               │
    ├── 落子无效? → 重新 request ────┘
    │
    ├── logic.place_stone(row, col)
    ├── emit stone_placed
    ├── 游戏结束? → emit game_ended
    │
    └── 未结束 → 切换回合 ───────────┘
```

## 4. AI 算法详解

### Level 1: 随机 AI

最简单的 AI。在已有棋子附近（Manhattan 距离 ≤ 2）的空位中随机选一个。

```
已有棋子: ●
搜索范围 (半径2):

  . . . . .
  . x x x .     x = 候选位置（空位）
  . x ● x .     随机选一个
  . x x x .
  . . . . .
```

**为什么不在全棋盘随机？** 因为远离已有棋子的位置几乎没有价值。限制范围让 AI
至少看起来在"下棋"而不是乱放。

---

### Level 2: 启发式评估 (Heuristic)

**核心思想：** 给每个空位打分，选分最高的。

打分方法：假设在该位置放一个子，看 4 个方向能形成什么棋型：

```
棋型评分表:
  ┌────────────┬──────────┬─────────────────────┐
  │   棋型      │   分数    │   图示                │
  ├────────────┼──────────┼─────────────────────┤
  │ 五连       │  100,000 │  ● ● ● ● ●          │
  │ 活四       │   10,000 │  . ● ● ● ● .        │
  │ 冲四       │    1,000 │  × ● ● ● ● .        │
  │ 活三       │    1,000 │  . ● ● ● .  .       │
  │ 眠三       │      100 │  × ● ● ● .          │
  │ 活二       │      100 │  . ● ● .  .          │
  │ 眠二       │       10 │  × ● ● .             │
  └────────────┴──────────┴─────────────────────┘
  × = 被堵住（边界或对手棋子）
  . = 空位

活四 vs 冲四:
  活四: 两端都开放 → 对手无法同时堵住 → 必胜
  冲四: 只有一端开放 → 对手可以堵 → 不一定赢
```

**攻防兼顾：** 每个空位同时计算进攻分（自己的棋型）和防守分（对方的棋型）。
防守权重略高（×1.1），让 AI 优先堵对方而不是自顾自进攻。

```
总分 = 进攻分 + 防守分 × 1.1

示例: 对手有活三，自己有活二
  防守分 = 1000 × 1.1 = 1100
  进攻分 = 100
  → 选择堵对方（1100 > 100）
```

### Level 3: Minimax + Alpha-Beta 剪枝

**核心思想：** "如果我走这步，对手最好的应对是什么？然后我再最好的应对......" 
递归博弈，看 N 步之后谁更有利。

#### Minimax 算法

```
我的视角(最大化):    对手视角(最小化):
         [?]                    ← 我选最大的
        / | \
      [?] [?] [?]              ← 对手选最小的
     /|   /|   |\
   [3][5] [2][4] [1][6]        ← 叶子节点：棋型评分

反向推导:
         [2]                    ← 我选 max(5, 2, 1) ... 不对
        / | \
      [5] [2] [1]              ← 对手选 min
     /|   /|   |\
   [3][5] [2][4] [1][6]

等等，让我重新画:

深度0 (我): 选最大值
   ├── 走法A → 深度1 (对手): 选最小值
   │   ├── 走法A1 → 评分 3
   │   └── 走法A2 → 评分 5
   │   对手会选 min(3,5) = 3
   ├── 走法B → 深度1 (对手): 选最小值
   │   ├── 走法B1 → 评分 2
   │   └── 走法B2 → 评分 8
   │   对手会选 min(2,8) = 2
   └── 走法C → 深度1 (对手): 选最小值
       ├── 走法C1 → 评分 7
       └── 走法C2 → 评分 4
       对手会选 min(7,4) = 4

我选 max(3, 2, 4) = 4 → 选走法C
```

#### Alpha-Beta 剪枝

Minimax 的问题：搜索树太大。15x15 棋盘，每步约 20-40 个候选，
深度 4 = 40^4 = 256 万个节点。太慢。

**Alpha-Beta 剪枝**可以跳过不需要搜索的分支：

```
alpha = 我已经找到的最好选择 (下界)
beta  = 对手已经找到的最好选择 (上界)

如果 alpha ≥ beta → 剪枝！不用继续搜了。

示例:
深度0 (我, alpha=-∞, beta=+∞):
   ├── 走法A → 对手选了 min = 3  → alpha 更新为 3
   ├── 走法B → 对手开始搜:
   │   ├── B1 → 评分 2  → 2 < alpha(3)
   │   └── 剪枝！不用看 B2 了
   │         因为对手最多给我 2 分，而我已经有 3 分的选择
   └── 走法C → ...

效果: 好的走法排序下，可以剪掉 90%+ 的节点
```

#### 候选走法生成

不搜全部 225 个空位，只搜已有棋子附近 2 格内的空位（约 20-40 个），
然后按启发式评分排序（高分优先），这样 Alpha-Beta 剪枝效率更高。

### Level 4: 迭代加深 + 置换表 + Killer Move

在 Level 3 基础上加三个优化：

#### 迭代加深 (Iterative Deepening)

```
先搜深度 1 → 找到最佳走法 M1
再搜深度 2 → 用 M1 作为第一个候选 → 找到 M2
再搜深度 3 → 用 M2 作为第一个候选 → 找到 M3
最终搜深度 4 → 用 M3 作为第一个候选 → 找到 M4

好处: 每一层的最佳走法给下一层提供极好的走法排序
     → Alpha-Beta 剪枝效率大幅提升
```

#### 置换表 (Transposition Table)

用 **Zobrist 哈希** 记录已经评估过的棋盘状态，避免重复计算。

```
Zobrist 哈希原理:

预生成随机数表: zobrist[15][15][3]  (每个位置每种棋子一个随机数)

棋盘哈希 = XOR(所有已放棋子的随机数)

放子: hash ^= zobrist[row][col][piece]    ← O(1)
撤子: hash ^= zobrist[row][col][piece]    ← 再 XOR 一次就还原

置换表: Dictionary { hash → {depth, score, flag, best_move} }
  命中 → 直接返回之前的结果，不用重新搜索
```

#### Killer Move 启发

```
记录每层搜索中引起 Beta 截断（剪枝）的走法。
下次搜同一层时，优先尝试这些 "杀手走法"。

原理: 在同一深度，好的走法往往在不同位置也是好的。
例如: 某个防守位置在多个变化中都能引起剪枝。
```

### Level 5: 蒙特卡洛树搜索 (MCTS)

> **2026-04-16 重写**：最初的 L5 是"UCB1 + 半随机 rollout"，实测 20 局对 L4 全败。
> 诊断：随机 rollout 的胜率信号噪声太大，在 3000 次模拟下都压不过 Minimax d4 的棋型评估。
> 现在的 L5 是 **PUCT + 棋型先验 + 棋型叶值**——把 AlphaZero 的搜索结构套在纯棋型评估上，
> 不需要神经网络也能产生强棋力。代码在 [`scripts/ai/ai_mcts.gd`](../scripts/ai/ai_mcts.gd)。

#### 四个阶段

```
重复 N 次 (N = 1500):

┌─────────────────────────────────────────────────┐
│ 1. 选择 (Selection)                              │
│    从根节点用 PUCT 公式一路往下，每一步都选当前       │
│    PUCT 值最高的子节点，直到走到叶节点。              │
├─────────────────────────────────────────────────┤
│ 2. 扩展 (Expansion)                              │
│    叶节点用棋型评估器 (pattern_evaluator) 给         │
│    所有候选走法打分，归一化成先验概率 P(a)，          │
│    一次性创建所有子节点。                           │
├─────────────────────────────────────────────────┤
│ 3. 叶值评估 (Leaf Evaluation)                     │
│    不再做随机 rollout。直接对叶节点棋盘做一次          │
│    evaluate_board(棋型总分差)，用 tanh 映射到        │
│    [0, 1] 区间作为本次模拟的"胜率"。                 │
│                                                  │
│    value = (tanh(score_diff / 1000) + 1) / 2     │
├─────────────────────────────────────────────────┤
│ 4. 反向传播 (Backpropagation)                     │
│    把 value 沿路径传回去，累加访问次数和 wins。       │
│    注意：每层按谁是"当前玩家"来决定 value/1-value。  │
└─────────────────────────────────────────────────┘

最终选择: 根的访问次数最多的子节点
```

#### PUCT 公式（AlphaZero 风格）

```
PUCT(child) = Q(child) + c_puct × P(child) × √(N_parent) / (1 + N_child)
              \_______/   \________________________________________/
               利用项                     探索项
           (子节点平均胜率)     (先验越高、被访问越少 → 越值得探索)

Q(child) = child.wins / child.visits    (平均胜率)
P(child) = 棋型评估归一化的先验概率
c_puct  = 2.5   (探索常数)

为什么不是 UCB1？
  - UCB1 的探索项只看访问次数，对每个走法一视同仁
  - PUCT 的探索项乘了 P(a)——"棋型评估说这个走法好"的先验
  - 15×15 的棋盘候选走法太多 (~100+)，没有先验的 MCTS
    会在废走法上浪费大量模拟
```

#### 即时战术短路

在创建根节点之前，先扫一遍候选走法：
1. **立即胜**：如果有一步能形成五连，直接下
2. **立即防**：如果对手下一步能成五，必须堵

这两个检查避免 MCTS 在"明显必下"局面浪费模拟。

#### 棋型叶值替代 rollout 的原因

原来的半随机 rollout（有活四必下、有活三必堵）看起来合理，
但实测下来对一盘终局还有 20 步的局面，rollout 结果基本是噪声——
两个"半智能"玩家随机走下来，输赢和当前局面好坏关系不大。

棋型评估器是确定性的：它直接报告"当前局面双方棋型总分差"，
`tanh(score_diff / 1000)` 给 MCTS 一个光滑的梯度信号，
同样数量的模拟下收敛明显更快。

#### MCTS vs Minimax

```
┌──────────────┬─────────────┬────────────────────┐
│              │  Minimax     │  MCTS (L5 当前版)   │
├──────────────┼─────────────┼────────────────────┤
│ 评估方式      │ 棋型总分      │ 棋型总分 (叶值)      │
│ 搜索方式      │ 穷举固定深度   │ PUCT 选择性展开      │
│ 先验信息      │ 无（纯排序）   │ 棋型先验引导         │
│ 深度          │ 固定 (2-4层) │ 不固定 (自适应)      │
│ 对复杂战术     │ 看到 4 层为止 │ 先验强→搜索集中        │
│ 对废招         │ 全部展开      │ 先验低→几乎不访问    │
│ 时间控制      │ 难 (固定深度) │ 易 (模拟次数可调)    │
└──────────────┴─────────────┴────────────────────┘
```

### Level 6: 神经网络 + MCTS (AlphaZero 风格)

**核心思想：** 用神经网络 + 战术搜索共同替代 MCTS 中的"弱环节"：
1. **走法选择** → 策略头 (Policy) 与棋型先验按 50/50 混合
2. **局面评估** → 连续棋型分差（L6 用 Hybrid 模式，CNN 价值头仅用于纯 CNN 模式）
3. **必杀检测** → VCF/VCT 战术搜索在每次 `mcts.search()` 前运行

```
L6 Hybrid 模式搜索流程（代码：ai_server/ai/mcts_engine.py）:

1. 战术短路（vcf_depth > 0 时启用）
   a. find_vcf(board, self)  → 如果有五连强制，直接返回
   b. find_vct(board, self)  → 如果有 3 步以内必杀威胁序列，直接返回
   c. find_vcf(board, opp)   → 对手有必杀？必须堵
   d. find_vct(board, opp)   → 对手 3 步必杀？必须堵

2. MCTS（simulations 次迭代，默认 200）
   - PUCT 选择: Q + c_puct × (0.5·P_CNN + 0.5·P_pattern) × √N / (1+n)
   - 叶值: _continuous_leaf_value = tanh(score_diff / 1000)
     （不是 value head——小网络 value head 太嘈杂）

3. 返回根访问分布作为策略，取 argmax 为最终落子

公式:
  传统:       UCB1  = Q + C × √(ln(N)/n)
  AlphaZero:  PUCT  = Q + C × P × √(N) / (1+n)
  L6 (Hybrid):  同 PUCT，P = 0.5·CNN + 0.5·pattern

simulations: 200 (2026-04-16 从 80 提升，用户反馈"感觉可以再慢点")
vcf_depth:   10  (VCF 最大深度)
vcf_branch:  6   (VCT 每步最大分支数)
```

#### 为什么 Hybrid 模式而不是纯 CNN？

小网络 (128f/6b, 1.4M 参数) 的 value head 评估叶节点噪声大，
直接用会让 MCTS 比纯棋型叶值还弱。混合方案：
- **Policy** 用 50% CNN + 50% 棋型先验——CNN 学到的抽象模式 + 可靠的战术底线
- **Value** 用棋型评估的 tanh 分差，不用 CNN value head
- **战术层** VCF/VCT 专门处理必杀，MCTS 不用为它操心

实测 CNN prior 权重 > 50% 会引起回归（75% 权重测过，分数跌到 12%）。

## 5. 棋型评估系统

### 连续棋型识别

对每个空位，在 4 个方向上扫描，统计连续同色子和开放端：

```
扫描示例 (水平方向):

  . . ○ ● ● [?] ● . ○ .
              ↑ 待评估位置

  向右: ● (1个连续) → 空位 → 停止
  向左: ● ● (2个连续) → ○ → 停止

  如果放黑子: 连续 = 2 + 1 + 1 = 4
  开放端: 右边开放(1), 左边被堵(0)
  → 冲四 (half_four, 1000分)
```

### 分裂棋型识别（2026-04-17 补）

> 最初版本的棋型评估器**只看连续棋型**，导致 AI 看不到 `XX_X` 型的分裂三、
> `XXX_X` 型的分裂四。用户老婆下出一组分裂四组合击败了 L6——赛后复盘时意识到
> "分裂三/分裂四本来就是标准棋型"，遂补上。

核心是 `_scan_line()`：除了向一侧扫描连续同色子，还**跨过一个空位继续扫描**，
把"缺口另一侧的同色子"也数进去。

```
扫描示例（带缺口）:

  . . ● ● [?] . ● ● .
              ↑ 待评估位置

  向右: 空位 → 停止连续
        连续同色 = 0
        缺口另一侧: ● ● (2 个)

  向左: ● ● → 连续 = 2

  棋型计分:
    • 连续棋型: 1 + 0 + 2 = 3 → open_three
    • 分裂棋型: 1 + 0 + 2 + 2 (缺口另一侧) = 5 → 但真实总长 4
      其中"能填缺口变成连五"的棋型 → 按 half_four 记

  最终分数: max(连续分, 分裂分)
```

#### 分裂棋型评分规则

- `N 个棋子 + 1 个缺口，N ≥ 4` → 填缺口即成五连 → 记为 **half_four** (1000)
- `N 个棋子 + 1 个缺口，N == 3` → 填缺口即成四 → 记为 **half_three** (100)
- 每个方向按 `max(连续分, 分裂分)` 取较大者累加

Python (`ai_server/ai/pattern_eval.py`) 和 GDScript (`scripts/ai/pattern_evaluator.gd`)
双侧都实现了分裂棋型检测，保证 L2/L4/L5 (GDScript) 和 L6 (Python) 行为一致。
对应的回归测试在 `tests/test_gapped_patterns.gd`。

### 双威胁加成（仅 Python 侧）

一步棋同时造成两个及以上"强棋型"（open_three 或更强）时，
`evaluate_position()` 会把累加分乘以 3 倍：

```
strong_count >= 2 时:
  total *= 3.0   # DOUBLE_THREAT_BONUS
```

这对"双活三"、"四三"之类**必胜组合**尤其重要——
普通按方向求和会让"一个很强的棋型"的分数盖过"两个中等棋型"，
但后者在实战中通常是杀棋。

### VCF / VCT 战术搜索

棋型评估只告诉你"这一步下去局面怎么样"，对**多步必杀**需要专门的战术搜索。

- **VCF (Victory by Continuous Fours)** — 连续冲四强制必杀，
  分支因子 1（每步只看冲四走法），深度可拉到 10+ 很快返回。
  代码：`ai_server/ai/vcf_search.py`。
- **VCT (Victory by Continuous Threats)** — 除了冲四，还允许"活三"作为威胁步，
  分支因子 2-6，深度 6 以内可实时返回。能捕捉"双三"、"冲四 + 活三"等杀棋。
  代码：`ai_server/ai/vct_search.py`（2026-04-16 补）。

L6 的 `mcts_engine.py` 在每次 `search()` 开头都跑一次 VCF + VCT（己方/对方各一次），
有命中就直接返回对应落子，跳过 MCTS。这让"人眼一看就是必杀"的局面得到秒解，
不被 MCTS 的随机性浪费时间。

## 6. 神经网络架构

### CNN 模型结构

```
输入: 2 × 15 × 15
  ┌───────────┐
  │ 通道0: 己方 │  1 = 有己方棋子, 0 = 没有
  │ 通道1: 对方 │  1 = 有对方棋子, 0 = 没有
  └─────┬─────┘
        ▼
  ┌─────────────────┐
  │ Conv2D(2→F, 3×3) │  F = filters (8 for tiny, 64 for full)
  │ + BatchNorm      │
  │ + ReLU           │
  └─────┬───────────┘
        ▼
  ┌─────────────────┐
  │  ResBlock × N    │  N = 1 (tiny) 或 3 (full)
  │                  │
  │  ┌─────────┐    │    ResBlock 内部:
  │  │Conv 3×3 │    │      输入 ──┐
  │  │BN + ReLU│    │      Conv → BN → ReLU
  │  │Conv 3×3 │    │      Conv → BN
  │  │BN       │    │      + 输入 (跳跃连接)
  │  │+ 跳跃连接│    │      ReLU
  │  │ReLU     │    │
  │  └─────────┘    │
  └─────┬───────────┘
        │
   ┌────┴────┐
   ▼         ▼
┌──────┐  ┌──────┐
│策略头 │  │价值头 │
│      │  │      │
│Conv1 │  │Conv1 │
│BN    │  │BN    │
│ReLU  │  │ReLU  │
│FC    │  │FC→64 │
│225维 │  │FC→1  │
│Softmax│ │Tanh  │
│      │  │      │
│概率   │  │评分  │
│[225] │  │[-1,1]│
└──────┘  └──────┘

策略头输出: 225 个概率值 → 每个位置被选为下一步的概率
价值头输出: 1 个数值 → 当前局面对当前玩家的评估
  +1 = 我赢定了
   0 = 势均力敌
  -1 = 我要输了
```

### 为什么是 CNN？

```
五子棋棋盘是 2D 网格，棋型是局部空间模式:

  活三:  . ● ● ● .      ← 水平模式
  活三:  .              ← 垂直模式
         ●
         ●
         ●
         .

CNN 的卷积核天然擅长识别这种空间模式！
3×3 卷积核扫过棋盘，能检测局部特征。
多层 CNN 能组合局部特征成为更复杂的全局理解。

这就是为什么 AlphaGo/AlphaZero 选择 CNN — 
棋盘是图像，下棋是图像识别问题。
```

### ResBlock (残差块) 的作用

```
普通网络:    x → Conv → Conv → Conv → 输出
问题: 太深的网络梯度会消失，学不动

残差网络:    x → Conv → Conv ──→ (+x) → 输出
                                  ↑
                              跳跃连接

好处:
  - 如果这两层 Conv 啥也没学到 → 至少输出 = 输入（不会更差）
  - 梯度可以通过跳跃连接直接回传 → 深层也能学
  - 使得训练更深的网络成为可能
```

### BatchNorm (批归一化) 的作用

```
每层输出的数值范围不一致会导致训练不稳定。

BatchNorm:
  输出 = (输入 - 均值) / √方差 × γ + β
         \___________________/   ↑   ↑
          归一化到 N(0,1)      缩放  偏移
                              (可学习参数)

效果: 稳定训练，加速收敛，允许更大学习率
```

## 7. 训练管线

### AlphaZero 训练循环

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  初始化: 随机权重的 CNN (啥也不会)                  │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │  循环 (每循环 = 一代):                       │  │
│  │                                            │  │
│  │  1. 自我对弈 (200 局)                       │  │
│  │     用当前 CNN + MCTS 下棋                   │  │
│  │     每步记录:                               │  │
│  │       - 棋盘状态 (15×15)                    │  │
│  │       - MCTS 搜索概率 (225维)               │  │
│  │       - 最终胜负 (+1/-1/0)                  │  │
│  │                                            │  │
│  │          ↓ ~20000 个训练样本                  │  │
│  │                                            │  │
│  │  2. 训练 CNN (10 epochs)                    │  │
│  │     损失 = 策略损失 + 价值损失               │  │
│  │                                            │  │
│  │     策略损失 = -Σ π_mcts × log(P_nn)       │  │
│  │     (让 NN 的输出趋近 MCTS 搜索概率)         │  │
│  │                                            │  │
│  │     价值损失 = (V_nn - z)²                  │  │
│  │     (让 NN 的评估趋近实际胜负)               │  │
│  │                                            │  │
│  │  3. 评估 (20 局)                            │  │
│  │     新模型 vs 旧模型                        │  │
│  │     胜率 > 55% → 保存为新一代               │  │
│  │     胜率 ≤ 55% → 丢弃，继续用旧模型         │  │
│  │                                            │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  结果: 每一代都比上一代更强                        │
│  第0代: 随机水平                                  │
│  第3代: 能看懂基本棋型                            │
│  第10代: 有策略性进攻防守                          │
└──────────────────────────────────────────────────┘
```

### 自我对弈为什么有效？

```
传统机器学习: 需要人类标注的数据
  "这步是好棋" "这步是坏棋" → 需要专家

AlphaZero: 不需要任何人类知识！
  1. AI 自己和自己下棋
  2. 赢了 → 过程中的走法被认为是"好的"
  3. 输了 → 过程中的走法被认为是"差的"
  4. CNN 学习区分好走法和差走法
  5. 下一轮自我对弈用更聪明的 CNN → 质量更高的数据
  6. 正反馈循环 → 越来越强

类比: 一个人关在房间里只有棋盘
  第1天: 随便放，偶尔赢了记住怎么走的
  第10天: 发现 "连成一排好像容易赢"
  第100天: 发现 "做活三逼对手堵"
  第1000天: 各种战术战略
```

### 训练参数说明

```python
python train_pipeline.py \
    --generations 10    # 训练 10 代
    --games 100         # 每代自我对弈 100 局
    --simulations 200   # 每步 MCTS 搜索 200 次
    --epochs 10         # 每代训练 10 轮
    --filters 8         # CNN 8 个卷积核 (tiny)
    --blocks 1          # 1 个残差块 (tiny)
    --lr 0.001          # 学习率 (Adam 优化器)
    --eval-games 20     # 新旧模型对战 20 局

# Tiny vs Full 模型:
#   Tiny (8, 1):  训练快，GDScript 可推理，棋力有限
#   Full (64, 3): 训练慢，需 Python 服务，棋力强
```

## 附录: 关键概念速查

| 概念 | 一句话解释 |
|------|----------|
| **Minimax** | 我选最好的，假设对手也选最好的 |
| **Alpha-Beta** | 跳过不可能更好的分支，加速 Minimax |
| **MCTS** | 通过大量随机模拟来估计走法质量 |
| **UCB1** | 平衡"利用已知好走法"和"探索未知走法" |
| **CNN** | 卷积神经网络，擅长识别 2D 空间模式 |
| **ResBlock** | 残差块，让深层网络也能训练 |
| **BatchNorm** | 归一化层，稳定训练 |
| **策略头** | CNN 输出：每个位置的落子概率 |
| **价值头** | CNN 输出：当前局面的胜负评估 |
| **Self-play** | AI 自己和自己下棋，生成训练数据 |
| **ONNX** | 模型交换格式，训练和部署解耦 |
| **Zobrist** | 用异或运算快速计算棋盘哈希值 |
| **置换表** | 缓存已评估的棋盘状态，避免重复计算 |
| **Killer Move** | 记录好的走法，下次优先尝试 |
