### 胜负判定

每次落子后，从该位置出发检查 4 个方向：

```
方向1: 水平   → (0, +1) 和 (0, -1)
方向2: 垂直   → (+1, 0) 和 (-1, 0)
方向3: 对角↘  → (+1, +1) 和 (-1, -1)
方向4: 对角↙  → (+1, -1) 和 (-1, +1)

示例: 黑子在 (7,7) 落子后检查水平方向
  ← 向左数连续黑子 + 1(自身) + 向右数连续黑子 ≥ 5 → 赢

  . . ● ● ● [●] ● . .     ← 5连，黑赢!
              ↑ 刚落的子
```

时间复杂度：O(1) — 只检查落子位置周围，不扫描整个棋盘。

## 3. Player 抽象层

### 设计模式

```
             GameManager
                 │
        _on_move_decided(row, col)
                 │
     ┌───────────┼───────────┐
     │           │           │
 HumanPlayer  AIPlayer   NetworkPlayer
     │           │           │
  鼠标点击    AI计算(Thread)  RPC网络
```

核心思想：**GameManager 不关心落子从哪来**。它只管理两个 PlayerController 实例，
调用 `request_move()` 然后等 `move_decided` 信号。这样添加新的落子来源（AI、网络、
甚至文件回放）都不需要改 GameManager 的代码。

### 游戏循环

```
start_game()
    │
    ▼
_request_current_move()  ←──────────┐
    │                               │
    ▼                               │
 [等待 move_decided 信号]            │
    │                               │
    ▼                               │
_on_move_decided(row, col)          │
    │                               │
    ├── 落子无效? → 重新 request ────┘
    │
    ├── logic.place_stone(row, col)
    ├── emit stone_placed
    ├── 游戏结束? → emit game_ended
    │
    └── 未结束 → 切换回合 ───────────┘
```

## 4. AI 算法详解

### Level 1: 随机 AI

最简单的 AI。在已有棋子附近（Manhattan 距离 ≤ 2）的空位中随机选一个。

```
已有棋子: ●
搜索范围 (半径2):

  . . . . .
  . x x x .     x = 候选位置（空位）
  . x ● x .     随机选一个
  . x x x .
  . . . . .
```

**为什么不在全棋盘随机？** 因为远离已有棋子的位置几乎没有价值。限制范围让 AI
至少看起来在"下棋"而不是乱放。

---

### Level 2: 启发式评估 (Heuristic)

**核心思想：** 给每个空位打分，选分最高的。

打分方法：假设在该位置放一个子，看 4 个方向能形成什么棋型：

```
棋型评分表:
  ┌────────────┬──────────┬─────────────────────┐
  │   棋型      │   分数    │   图示                │
  ├────────────┼──────────┼─────────────────────┤
  │ 五连       │  100,000 │  ● ● ● ● ●          │
  │ 活四       │   10,000 │  . ● ● ● ● .        │
  │ 冲四       │    1,000 │  × ● ● ● ● .        │
  │ 活三       │    1,000 │  . ● ● ● .  .       │
  │ 眠三       │      100 │  × ● ● ● .          │
  │ 活二       │      100 │  . ● ● .  .          │
  │ 眠二       │       10 │  × ● ● .             │
  └────────────┴──────────┴─────────────────────┘
  × = 被堵住（边界或对手棋子）
  . = 空位

活四 vs 冲四:
  活四: 两端都开放 → 对手无法同时堵住 → 必胜
  冲四: 只有一端开放 → 对手可以堵 → 不一定赢
```

**攻防兼顾：** 每个空位同时计算进攻分（自己的棋型）和防守分（对方的棋型）。
防守权重略高（×1.1），让 AI 优先堵对方而不是自顾自进攻。

```
总分 = 进攻分 + 防守分 × 1.1

示例: 对手有活三，自己有活二
  防守分 = 1000 × 1.1 = 1100
  进攻分 = 100
  → 选择堵对方（1100 > 100）
```

### Level 3: Minimax + Alpha-Beta 剪枝

**核心思想：** "如果我走这步，对手最好的应对是什么？然后我再最好的应对......" 
递归博弈，看 N 步之后谁更有利。

#### Minimax 算法

```
我的视角(最大化):    对手视角(最小化):
         [?]                    ← 我选最大的
        / | \
      [?] [?] [?]              ← 对手选最小的
     /|   /|   |\
   [3][5] [2][4] [1][6]        ← 叶子节点：棋型评分

反向推导:
         [2]                    ← 我选 max(5, 2, 1) ... 不对
        / | \
      [5] [2] [1]              ← 对手选 min
     /|   /|   |\
   [3][5] [2][4] [1][6]

等等，让我重新画:

深度0 (我): 选最大值
   ├── 走法A → 深度1 (对手): 选最小值
   │   ├── 走法A1 → 评分 3
   │   └── 走法A2 → 评分 5
   │   对手会选 min(3,5) = 3
   ├── 走法B → 深度1 (对手): 选最小值
   │   ├── 走法B1 → 评分 2
   │   └── 走法B2 → 评分 8
   │   对手会选 min(2,8) = 2
   └── 走法C → 深度1 (对手): 选最小值
       ├── 走法C1 → 评分 7
       └── 走法C2 → 评分 4
       对手会选 min(7,4) = 4

我选 max(3, 2, 4) = 4 → 选走法C
```

#### Alpha-Beta 剪枝

Minimax 的问题：搜索树太大。15x15 棋盘，每步约 20-40 个候选，
深度 4 = 40^4 = 256 万个节点。太慢。

**Alpha-Beta 剪枝**可以跳过不需要搜索的分支：

```
alpha = 我已经找到的最好选择 (下界)
beta  = 对手已经找到的最好选择 (上界)

如果 alpha ≥ beta → 剪枝！不用继续搜了。

示例:
深度0 (我, alpha=-∞, beta=+∞):
   ├── 走法A → 对手选了 min = 3  → alpha 更新为 3
   ├── 走法B → 对手开始搜:
   │   ├── B1 → 评分 2  → 2 < alpha(3)
   │   └── 剪枝！不用看 B2 了
   │         因为对手最多给我 2 分，而我已经有 3 分的选择
   └── 走法C → ...

效果: 好的走法排序下，可以剪掉 90%+ 的节点
```

#### 候选走法生成

不搜全部 225 个空位，只搜已有棋子附近 2 格内的空位（约 20-40 个），
然后按启发式评分排序（高分优先），这样 Alpha-Beta 剪枝效率更高。

### Level 4: 迭代加深 + 置换表 + Killer Move

在 Level 3 基础上加三个优化：

#### 迭代加深 (Iterative Deepening)

```
先搜深度 1 → 找到最佳走法 M1
再搜深度 2 → 用 M1 作为第一个候选 → 找到 M2
再搜深度 3 → 用 M2 作为第一个候选 → 找到 M3
最终搜深度 4 → 用 M3 作为第一个候选 → 找到 M4

好处: 每一层的最佳走法给下一层提供极好的走法排序
     → Alpha-Beta 剪枝效率大幅提升
```

#### 置换表 (Transposition Table)

用 **Zobrist 哈希** 记录已经评估过的棋盘状态，避免重复计算。

```
Zobrist 哈希原理:

预生成随机数表: zobrist[15][15][3]  (每个位置每种棋子一个随机数)

棋盘哈希 = XOR(所有已放棋子的随机数)

放子: hash ^= zobrist[row][col][piece]    ← O(1)
撤子: hash ^= zobrist[row][col][piece]    ← 再 XOR 一次就还原

置换表: Dictionary { hash → {depth, score, flag, best_move} }
  命中 → 直接返回之前的结果，不用重新搜索
```

#### Killer Move 启发

```
记录每层搜索中引起 Beta 截断（剪枝）的走法。
下次搜同一层时，优先尝试这些 "杀手走法"。

原理: 在同一深度，好的走法往往在不同位置也是好的。
例如: 某个防守位置在多个变化中都能引起剪枝。
```

### Level 5: 蒙特卡洛树搜索 (MCTS)

**核心思想完全不同于 Minimax。** MCTS 不需要评估函数，
它通过大量随机对局来估计每个走法的胜率。

#### 四个阶段

```
重复 N 次 (N = 3000-5000):

┌─────────────────────────────────────────────────┐
│ 1. 选择 (Selection)                              │
│    从根节点开始，用 UCB1 公式选子节点，             │
│    直到找到一个有未尝试走法的节点。                  │
│                                                  │
│              Root                                │
│             / | \                                │
│           A   B   C    ← UCB1 选 B               │
│          / \   |                                 │
│         D   E  F       ← UCB1 选 F               │
│                |                                 │
│               [?]      ← F 还有未尝试的走法        │
├─────────────────────────────────────────────────┤
│ 2. 扩展 (Expansion)                              │
│    从未尝试的走法中随机选一个，创建新节点 G。        │
│                                                  │
│               F                                  │
│              / \                                 │
│             G  [?] [?]                           │
│             ↑ 新节点                              │
├─────────────────────────────────────────────────┤
│ 3. 模拟 (Simulation / Rollout)                   │
│    从 G 的状态开始，双方随机下棋直到分出胜负。       │
│                                                  │
│    G → 随机走 → 随机走 → ... → 黑赢!              │
│                                                  │
│    (我们的 rollout 不是纯随机，                     │
│     有活四必下，有活三必堵 → "半随机")              │
├─────────────────────────────────────────────────┤
│ 4. 反向传播 (Backpropagation)                     │
│    把结果沿路径传回去，更新所有经过节点的统计。       │
│                                                  │
│    Root (visits+1)                               │
│      └── B (visits+1)                            │
│           └── F (visits+1)                       │
│                └── G (visits+1, wins+1 if won)   │
└─────────────────────────────────────────────────┘

最终选择: 访问次数最多的子节点（最稳定的走法）
```

#### UCB1 公式

```
UCB1(node) = wins/visits + C × √(ln(parent.visits) / visits)
             \_________/   \________________________________/
              利用项              探索项
          (胜率高的优先)      (访问少的优先尝试)

C = √2 ≈ 1.414 (探索常数，平衡利用与探索)

直觉:
  - 胜率高的节点：利用项大 → 倾向选它
  - 很少被访问的节点：探索项大 → 也值得试试
  - 随着模拟次数增加，逐渐收敛到最优走法
```

#### MCTS vs Minimax

```
┌──────────────┬─────────────┬─────────────────┐
│              │  Minimax     │  MCTS           │
├──────────────┼─────────────┼─────────────────┤
│ 评估方式      │ 棋型评分函数  │ 随机模拟胜率      │
│ 搜索方式      │ 穷举所有变化  │ 统计抽样          │
│ 需要评估函数   │ 是           │ 否 (但有更好)     │
│ 深度          │ 固定 (2-4层) │ 不固定 (自适应)   │
│ 对未知局面     │ 依赖评估准确性│ 能发现意外走法    │
│ 时间控制      │ 难 (固定深度) │ 易 (停止模拟即可) │
│ 棋力          │ 强但有天花板  │ 更灵活            │
└──────────────┴─────────────┴─────────────────┘
```

### Level 6: 神经网络 + MCTS (AlphaZero 风格)

**核心思想：** 用神经网络替代 MCTS 中的两个"弱环节"：
1. **走法选择** → 用策略头 (Policy Head) 引导搜索方向
2. **局面评估** → 用价值头 (Value Head) 替代随机 rollout

```
传统 MCTS:
  选择: UCB1 (均匀先验)
  评估: 随机 rollout → 嘈杂、不准确

AlphaZero MCTS:
  选择: UCB1 + 神经网络策略 (告诉 MCTS 哪些走法值得搜)
  评估: 神经网络价值 → 快速、准确

公式变化:
  传统:  UCB1 = wins/visits + C × √(ln(N)/visits)
  Alpha: UCB1 = wins/visits + C × P(move) × √(N) / (1 + visits)
                                  ↑ 神经网络给的先验概率
```

## 5. 棋型评估系统

### 棋型识别方法

对每个空位，在 4 个方向上扫描，统计连续同色子和开放端：

```
扫描示例 (水平方向):

  . . ○ ● ● [?] ● . ○ .
              ↑ 待评估位置

  向右: ● (1个连续) → 空位 → 停止
  向左: ● ● (2个连续) → ○ → 停止

  如果放黑子: 连续 = 2 + 1 + 1 = 4
  开放端: 右边开放(1), 左边被堵(0)
  → 冲四 (half_four, 1000分)
```

### 权重自我调优

```
训练流程:
  1. AI vs AI 批量对弈 100 局
  2. 统计每种棋型在胜方/败方的出现频率
  3. 胜方高频棋型 → 权重上调
     败方高频棋型 → 权重下调
  4. 新权重 vs 旧权重对弈验证
  5. 新权重胜率 > 55% → 保存

公式:
  new_weight = old_weight × (1 + adjustment × learning_rate)
  learning_rate = 0.02 (保守调整，避免过拟合)
```

## 6. 神经网络架构

### CNN 模型结构

```
输入: 2 × 15 × 15
  ┌───────────┐
  │ 通道0: 己方 │  1 = 有己方棋子, 0 = 没有
  │ 通道1: 对方 │  1 = 有对方棋子, 0 = 没有
  └─────┬─────┘
        ▼
  ┌─────────────────┐
  │ Conv2D(2→F, 3×3) │  F = filters (8 for tiny, 64 for full)
  │ + BatchNorm      │
  │ + ReLU           │
  └─────┬───────────┘
        ▼
  ┌─────────────────┐
  │  ResBlock × N    │  N = 1 (tiny) 或 3 (full)
  │                  │
  │  ┌─────────┐    │    ResBlock 内部:
  │  │Conv 3×3 │    │      输入 ──┐
  │  │BN + ReLU│    │      Conv → BN → ReLU
  │  │Conv 3×3 │    │      Conv → BN
  │  │BN       │    │      + 输入 (跳跃连接)
  │  │+ 跳跃连接│    │      ReLU
  │  │ReLU     │    │
  │  └─────────┘    │
  └─────┬───────────┘
        │
   ┌────┴────┐
   ▼         ▼
┌──────┐  ┌──────┐
│策略头 │  │价值头 │
│      │  │      │
│Conv1 │  │Conv1 │
│BN    │  │BN    │
│ReLU  │  │ReLU  │
│FC    │  │FC→64 │
│225维 │  │FC→1  │
│Softmax│ │Tanh  │
│      │  │      │
│概率   │  │评分  │
│[225] │  │[-1,1]│
└──────┘  └──────┘

策略头输出: 225 个概率值 → 每个位置被选为下一步的概率
价值头输出: 1 个数值 → 当前局面对当前玩家的评估
  +1 = 我赢定了
   0 = 势均力敌
  -1 = 我要输了
```

### 为什么是 CNN？

```
五子棋棋盘是 2D 网格，棋型是局部空间模式:

  活三:  . ● ● ● .      ← 水平模式
  活三:  .              ← 垂直模式
         ●
         ●
         ●
         .

CNN 的卷积核天然擅长识别这种空间模式！
3×3 卷积核扫过棋盘，能检测局部特征。
多层 CNN 能组合局部特征成为更复杂的全局理解。

这就是为什么 AlphaGo/AlphaZero 选择 CNN — 
棋盘是图像，下棋是图像识别问题。
```

### ResBlock (残差块) 的作用

```
普通网络:    x → Conv → Conv → Conv → 输出
问题: 太深的网络梯度会消失，学不动

残差网络:    x → Conv → Conv ──→ (+x) → 输出
                                  ↑
                              跳跃连接

好处:
  - 如果这两层 Conv 啥也没学到 → 至少输出 = 输入（不会更差）
  - 梯度可以通过跳跃连接直接回传 → 深层也能学
  - 使得训练更深的网络成为可能
```

### BatchNorm (批归一化) 的作用

```
每层输出的数值范围不一致会导致训练不稳定。

BatchNorm:
  输出 = (输入 - 均值) / √方差 × γ + β
         \___________________/   ↑   ↑
          归一化到 N(0,1)      缩放  偏移
                              (可学习参数)

效果: 稳定训练，加速收敛，允许更大学习率
```

## 7. 训练管线

### AlphaZero 训练循环

```
┌──────────────────────────────────────────────────┐
│                                                  │
│  初始化: 随机权重的 CNN (啥也不会)                  │
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │  循环 (每循环 = 一代):                       │  │
│  │                                            │  │
│  │  1. 自我对弈 (200 局)                       │  │
│  │     用当前 CNN + MCTS 下棋                   │  │
│  │     每步记录:                               │  │
│  │       - 棋盘状态 (15×15)                    │  │
│  │       - MCTS 搜索概率 (225维)               │  │
│  │       - 最终胜负 (+1/-1/0)                  │  │
│  │                                            │  │
│  │          ↓ ~20000 个训练样本                  │  │
│  │                                            │  │
│  │  2. 训练 CNN (10 epochs)                    │  │
│  │     损失 = 策略损失 + 价值损失               │  │
│  │                                            │  │
│  │     策略损失 = -Σ π_mcts × log(P_nn)       │  │
│  │     (让 NN 的输出趋近 MCTS 搜索概率)         │  │
│  │                                            │  │
│  │     价值损失 = (V_nn - z)²                  │  │
│  │     (让 NN 的评估趋近实际胜负)               │  │
│  │                                            │  │
│  │  3. 评估 (20 局)                            │  │
│  │     新模型 vs 旧模型                        │  │
│  │     胜率 > 55% → 保存为新一代               │  │
│  │     胜率 ≤ 55% → 丢弃，继续用旧模型         │  │
│  │                                            │  │
│  └────────────────────────────────────────────┘  │
│                                                  │
│  结果: 每一代都比上一代更强                        │
│  第0代: 随机水平                                  │
│  第3代: 能看懂基本棋型                            │
│  第10代: 有策略性进攻防守                          │
└──────────────────────────────────────────────────┘
```

### 自我对弈为什么有效？

```
传统机器学习: 需要人类标注的数据
  "这步是好棋" "这步是坏棋" → 需要专家

AlphaZero: 不需要任何人类知识！
  1. AI 自己和自己下棋
  2. 赢了 → 过程中的走法被认为是"好的"
  3. 输了 → 过程中的走法被认为是"差的"
  4. CNN 学习区分好走法和差走法
  5. 下一轮自我对弈用更聪明的 CNN → 质量更高的数据
  6. 正反馈循环 → 越来越强

类比: 一个人关在房间里只有棋盘
  第1天: 随便放，偶尔赢了记住怎么走的
  第10天: 发现 "连成一排好像容易赢"
  第100天: 发现 "做活三逼对手堵"
  第1000天: 各种战术战略
```

### 训练参数说明

```python
python train_pipeline.py \
    --generations 10    # 训练 10 代
    --games 100         # 每代自我对弈 100 局
    --simulations 200   # 每步 MCTS 搜索 200 次
    --epochs 10         # 每代训练 10 轮
    --filters 8         # CNN 8 个卷积核 (tiny)
    --blocks 1          # 1 个残差块 (tiny)
    --lr 0.001          # 学习率 (Adam 优化器)
    --eval-games 20     # 新旧模型对战 20 局

# Tiny vs Full 模型:
#   Tiny (8, 1):  训练快，GDScript 可推理，棋力有限
#   Full (64, 3): 训练慢，需 Python 服务，棋力强
```

## 附录: 关键概念速查

| 概念 | 一句话解释 |
|------|----------|
| **Minimax** | 我选最好的，假设对手也选最好的 |
| **Alpha-Beta** | 跳过不可能更好的分支，加速 Minimax |
| **MCTS** | 通过大量随机模拟来估计走法质量 |
| **UCB1** | 平衡"利用已知好走法"和"探索未知走法" |
| **CNN** | 卷积神经网络，擅长识别 2D 空间模式 |
| **ResBlock** | 残差块，让深层网络也能训练 |
| **BatchNorm** | 归一化层，稳定训练 |
| **策略头** | CNN 输出：每个位置的落子概率 |
| **价值头** | CNN 输出：当前局面的胜负评估 |
| **Self-play** | AI 自己和自己下棋，生成训练数据 |
| **ONNX** | 模型交换格式，训练和部署解耦 |
| **Zobrist** | 用异或运算快速计算棋盘哈希值 |
| **置换表** | 缓存已评估的棋盘状态，避免重复计算 |
| **Killer Move** | 记录好的走法，下次优先尝试 |
